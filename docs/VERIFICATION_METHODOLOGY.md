# ExecFlex Verified — Verification Methodology

**Date:** 2026-07-04
**Status:** Design + scaffold (assessment engine not built)

---

## What "ExecFlex Verified" Means

An "ExecFlex Verified" badge on a candidate profile means the candidate has passed a structured, third-party assessment validating specific competencies relevant to a talent pool category.

**Verified does NOT mean:**
- ExecFlex endorses the candidate's overall ability
- The candidate is guaranteed to be a good hire
- The assessment is a substitute for interviews or reference checks

**Verified DOES mean:**
- The candidate completed a standardized assessment from a recognized provider
- The assessment was administered under controlled conditions
- The candidate's score meets or exceeds the pool's minimum threshold
- The assessment result is on record and auditable

## Pool Categories and Assessment Types

| Pool Category | Assessment Domain | Example Providers | Minimum Score |
|--------------|-------------------|-------------------|---------------|
| **Tech — Software Engineering** | Algorithmic problem solving, system design, language proficiency | Codility, HackerRank, CodeSignal | 70/100 |
| **Tech — Data Science** | Statistical analysis, ML fundamentals, Python/R proficiency | Codility, DataCamp Assessments | 70/100 |
| **Finance — CFO/FD** | Financial analysis, forecasting, IFRS/GAAP, risk management | SHL, Korn Ferry, custom rubric | 65/100 |
| **Finance — Audit** | Internal controls, regulatory compliance, analytical reasoning | SHL, custom rubric | 65/100 |
| **Leadership** | Strategic thinking, team management, communication, decision-making | SHL OPQ, Hogan, DDI | 60/100 |
| **General Professional** | Cognitive ability, verbal/numerical reasoning | SHL Verify, Criteria Corp | 60/100 |

## Assessment Process

### Step 1: Pool Definition
An agency (org owner/recruiter) creates a talent pool with:
- Name and description
- `is_verified = true`
- `verification_method` reference to this document
- `criteria` JSON specifying: assessment provider, test ID, minimum score

### Step 2: Candidate Invitation
When a candidate is added to a verified pool:
1. The assessment adapter creates an assessment invitation via the provider's API
2. The candidate receives a link to complete the assessment
3. The assessment is timed and proctored per the provider's standards

### Step 3: Result Retrieval
After the candidate completes the assessment:
1. The adapter polls or receives a webhook with the result
2. `talent_pool_members.verified` is set based on score >= minimum threshold
3. `assessment_score`, `assessment_provider`, `assessment_data` are recorded
4. `verified_at` timestamp is set

### Step 4: Badge Display
- Verified candidates display the "ExecFlex Verified" badge in the UI
- The badge links to this methodology document
- Clients can click through to see: pool name, assessment provider, score range (not exact score), verified date

## Adapter Interface

```python
class AssessmentAdapter(Protocol):
    provider_name: str

    def create_assessment(self, candidate_id, pool_id, criteria) -> str:
        """Create an assessment invitation. Returns assessment URL/ID."""

    def get_result(self, assessment_id) -> Optional[AssessmentResult]:
        """Fetch the result of a completed assessment."""

    def list_available_tests(self) -> List[Dict[str, str]]:
        """List available test types from this provider."""
```

The interface is implemented at `services/talent_pools/assessment_adapter.py`. A `StubAssessmentAdapter` is provided for local dev and testing.

## Integrity Controls

1. **No self-assessment.** Candidates cannot verify themselves. Assessment links are generated by the system and administered by the provider.

2. **Score immutability.** Once an assessment result is recorded, it cannot be edited through the UI. Corrections require a new assessment.

3. **Expiry.** Verified status has a configurable TTL (default: 12 months). After expiry, the badge is removed and re-assessment is required.

4. **Audit trail.** All verification events (invitation sent, result received, badge granted/revoked) are logged in `activity_log`.

5. **Provider transparency.** The assessment provider name and test category are visible to clients viewing the badge. We do not obscure which tool was used.

## Defending the "Verified" Claim

If a client or candidate challenges a verified badge:

1. **Produce the assessment record** from `talent_pool_members` — shows provider, score, date.
2. **Produce the methodology document** (this file) — shows minimum thresholds.
3. **Produce the activity log** — shows when the assessment was created and completed.
4. **Contact the provider** if needed — assessment providers maintain their own audit trails.

The claim is defensible if and only if:
- A recognized provider administered the test
- The candidate's score met the stated threshold
- The assessment was completed within the last 12 months
- No manual override was applied to the `verified` flag

---

## Limitations

- **Assessments test skills, not character.** A verified engineer can still be a bad cultural fit.
- **Provider quality varies.** We select providers based on industry reputation, but we don't independently validate their psychometric properties.
- **Cheating is possible.** Remote proctoring reduces but does not eliminate cheating risk. This is a known limitation of all online assessment platforms.
- **Not built yet.** The adapter interface is scaffolded; no real provider integrations exist in v1. All assessment results in dev are from `StubAssessmentAdapter`.
